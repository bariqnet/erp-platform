# Changelog

All notable changes per phase. Conventional-commit style; entries land alongside the PR that introduces them.

## Phase 1 — Metadata Core

### Unreleased

- **TASK-01 · Monorepo Scaffold** — empty monorepo with tooling wired up, CLAUDE.md + RFC-001 committed, `pnpm verify` runs a green smoke test per package, CI runs `pnpm verify` on every push and PR.
- **TASK-02 · Local Dev Environment** — `infra/docker/compose.dev.yml` runs Postgres 16, Redis 7, OpenSearch 2 (warm ~10 s, healthy via `--wait`); `.env.example` documents every variable the stack and future services read; `docs/runbooks/local-dev.md` covers start/inspect/reset/teardown with verified commands; Testcontainers smoke suite in `packages/db` exercises all three services (6 tests / ~10 s); `pnpm test:integration` split from `pnpm test` so integration tests stay off the every-commit verify loop.
- **TASK-03 · Metadata Schema** — four RFC §4.1 tables (`metadata.meta_object`, `meta_change_set`, `meta_layer_activation`, `meta_audit_log`) with Kysely types, the `idx_meta_object_current` partial index, and RLS policies (ENABLED + FORCEd). A non-superuser `erp_app` role + `withTenantContext(db, tenantId, fn)` API together make policies actually fire in dev. SQL-file migration runner via Kysely's `Migrator` (idempotent, re-run-safe) plus a `pnpm db:migrate` CLI. Integration tests prove cross-tenant reads are blocked and the migrator round-trips cleanly. [ADR-0001](./docs/adr/0001-metadata-schema.md) records the non-DDL decisions; `docs/patterns/writing-a-migration.md` now has the real example.
- **TASK-04 · Core Domain Types** — `@erp/core` now exports Zod schemas with inferred TypeScript types for every metadata object type from RFC §2: `Layer`, `ObjectType`, `ObjectId`, `Locale`, `LocalizedString`, `LocalizationBody`, 15-variant discriminated `Field`, `Relationship`, `EntityBody` (with `.superRefine` rules for duplicate field names and money-field → currency-field reference), `PermissionBody`, plus the operation-discriminated `Envelope` (upsert | tombstone) and `envelopeWithBody<T>` helper for strongly-typed wrappers. 158 tests — every schema has a round-trip test proving `parse → JSON.stringify → JSON.parse → parse` is the identity function. `@erp/core`'s only production dep stays `zod`. `docs/patterns/writing-a-domain-type.md` populated with the real `LocalizedString`/`EntityBody` examples.
- **TASK-06 · Resolution Algorithm** — `@erp/metadata` now ships `resolve()` matching RFC §3.2 pseudocode exactly: walks layers bottom-up, deep-clones the first body, merges subsequent bodies per the candidate's strategy, resets effective + provenance on tombstone. Returns `Result<ResolvedObject<T>, { kind: "object_not_found"; ... }>`. Four pure merge strategies (`replace`, `merge_object`, `append`, `merge_list_by_key`) plus a `merge()` dispatcher. Zero I/O — the `MetadataStore` port lives in `@erp/core/src/ports` with `LayerCandidate` and `MergeStrategy` types. Property tests with fast-check verify determinism, tombstone correctness, and ordered layer application (100 runs each). Benchmark test asserts p99 < 0.5 ms for a 30-field × 3-layer resolve (measured: p99 ≈ 0.3 ms, p50 ≈ 0.08 ms). 38 metadata tests total.
- **TASK-05 · Result Type and EventBus Port** — `@erp/core` now exports `Result<T, E>` (plain discriminated-union data + namespace object with `ok`, `err`, `isOk`, `isErr`, `map`, `mapErr`, `flatMap`, `match`, `unwrap`, `unwrapOr`, `unwrapOrElse`) plus the `EventBus` port: Zod `DomainEventSchema` with UUID v4 event_id, dotted-lowercase event_type, W3C `TraceContext`, nullable tenant_id/actor_id, dedup_key, and a generic `DomainEvent<TPayload>` shape. `publish`/`subscribe`/`waitFor` interface + `Subscription` shape. 200 tests across `@erp/core` (+42 this task). `docs/patterns/handling-errors.md` populated with the real Result example, combinator table, HTTP-edge conversion, and the three-and-only-three cases where `throw` is correct.
- **TASK-07 · Change Set State Machine** — RFC §9.3 lifecycle implemented across two packages. `@erp/change-set` ships the pure pieces: state machine (`transition(from, action, actor)` returning `Result<NextState, TransitionError>` with role guards `metadata.write` / `metadata.approve` / `metadata.deploy`), Zod-discriminated `Operation` schemas (`upsert` / `tombstone`), and `buildChangeSetEvent()` factories per transition. `@erp/db` ships `ChangeSetRepository` extending the new `TenantRepository` base — atomic deploy materializes staged operations into `meta_object` rows inside one transaction (CHECK violation mid-batch rolls back every row), O(1) rollback via `superseded_by_change_set_id` (two `UPDATE`s bounded by the rollback's change_set_id). Migration 0002 adds `staged_operations jsonb` and the `superseded_by_change_set_id text` column. Audit entries write inside the same transaction as the data they audit. 20 integration tests in `@erp/db` (was 15) cover approve-then-deploy, deploy-then-rollback, and "failed deploy leaves system consistent". 58 unit tests in `@erp/change-set`.
- **TASK-08 · In-Process EventBus + Outbox** — `@erp/events` ships `OutboxBus` (implements the `EventBus` port from `@erp/core`) backed by `metadata.meta_outbox` (Migration 0003: UNIQUE on event_id + dedup_key, partial index on pending rows, RLS ENABLED + FORCED). Two publish surfaces: `publish()` (own tx) and `publishWithin(trx, event)` (atomic with caller's tx — the outbox-with-data pattern). `OutboxPump.drainOnce()` reads pending rows with `FOR UPDATE SKIP LOCKED`, dispatches via the bus, flips delivered_at on success / increments attempt_count on throw. Dead-letter cap at 8 attempts via `pump.deadLettered()`. `start()` / `stop()` for the polling loop with adaptive backoff. `apps/worker` exposes `createWorker(config)` that constructs DB + bus + pump in one wiring location per CLAUDE.md §7 #11. 7 integration tests cover atomicity, dedup_key collapse, pump delivery, restart durability, failing-handler retry, and per-type fan-out. `docs/patterns/emitting-an-event.md` populated with the two publish surfaces, idempotency rules, and pump internals.
- **TASK-09 · Fastify API Skeleton** — `apps/api` boots a Fastify 4 server via `buildServer()` (the single wiring point per CLAUDE.md §7 #11). Five plugins layered: telemetry (pino child + x-request-id + W3C trace*id), errors (RFC 7807 problem+json for ZodError, FST_ERR_VALIDATION, statusCode passthrough, 404 not-found), auth (placeholder for Better Auth — accepts x-user-id / x-user-roles in dev; full auth lands in TASK-10), tenant-context (validates x-tenant-id against `t*[a-z0-9_]{2,62}`, child-binds the logger), openapi (serves `/docs/openapi.json`from the registry every route registers against). Routes shipped:`/healthz`, `/readyz`(DB ping),`/docs/openapi.json`. Zod helpers `parseBody/parseQuery/parseParams`(Fastify 4 doesn't natively understand Zod; type-provider-zod requires Fastify 5). 12 contract tests via`fastify.inject()`cover every plugin path. Deps added to apps/api:`fastify@4.29.0`, `fastify-plugin@5.0.1`, `pino@catalog`, `@asteasolutions/zod-to-openapi@7.3.0`. `@erp/telemetry`ships`createLogger()`with structured fields + redaction (5 unit tests).`@erp/config/eslintrc.cjs`grows a`**/plugins/**/\*.ts`override so Fastify's`export default fp(...)`idiom keeps working without dropping no-default-export elsewhere.`docs/patterns/writing-a-route.md`populated with the canonical`/healthz` example + a tenant-scoped CRUD shape.
- **TASK-10 · Admin API for Metadata CRUD** — All 9 RFC §9.1 endpoints in `apps/api/src/routes/admin/`. Read-side: `GET /admin/v1/metadata/objects[?type=&layer=]`, `GET /admin/v1/metadata/objects/{id}` (resolved through `@erp/metadata`), `GET /admin/v1/metadata/objects/{id}/history` (tenant + vendor unioned). Write-side, role-gated via a new `requireRole(name)` preHandler factory: `POST /admin/v1/metadata/changes` (`metadata.write`), `/{id}/{simulate,propose}` (`metadata.write`), `/approve` (`metadata.approve`), `/{deploy,rollback}` (`metadata.deploy`). `@erp/db` ships `MetadataObjectRepository` extending `TenantRepository` and implementing the `MetadataStore` port (the resolver reads through it). Two thin services (`MetadataObjectService`, `ChangeSetService`) return `Result<T, ServiceError>`; routes Result.match into RFC 7807 problem+json with exhaustive error mapping. `@fastify/rate-limit` configured globally with per-tenant keys + per-route overrides on writes (60/min; simulate 30/min). `apps/api/src/openapi-registry.ts` calls `extendZodWithOpenApi(z)` so the generator can introspect parameter schemas. 16 contract tests via `fastify.inject()` cover the full lifecycle (propose → approve → deploy → rollback round-trips through the resolver). `docs/patterns/writing-a-service.md` populated with the real ChangeSetService example. **Better Auth deferred** — placeholder `x-user-id` / `x-user-roles` auth from TASK-09 gates role checks until TASK-10.1 lands. See [ADR-0002](./docs/adr/0002-better-auth-zod-4-deferral.md) for the Zod 4 migration plan.
- **TASK-11 · Application Kernel Service** — `apps/kernel` boots a standalone Fastify 4 service via `buildKernel()` (the single wiring point; CLAUDE.md §7 #11). `KernelCache` = in-process L1 Map-based LRU + optional ioredis L2 (keys `erp:kernel:<tenant>::<objectId>`, 1 h TTL), with a prefix-SCAN tenant sweep on invalidation and graceful degradation when Redis is unreachable. `CacheInvalidator` is a read-only outbox poller — initializes its cursor to `max(outbox_pk)` at boot, ticks every 250 ms, consumes only `metadata.change_set_deployed` events, and calls `cache.invalidateTenant(row.tenant_id)`. Each kernel instance runs its own; no cross-kernel coordination. The worker app's `OutboxPump` still owns flipping `delivered_at`. `ResolveService` is the hot path: probe cache → on miss, call `@erp/metadata.resolve()` through `MetadataObjectRepository` (which implements `MetadataStore`), store in both tiers, log `{event: "resolve", tenant_id, object_id, cache_status, duration_ms}`. Every resolve opens an OTel span (no-op until an SDK registers) — `@erp/telemetry` now exports `createTracer(scope)` backed by `@opentelemetry/api`. `POST /internal/resolve` returns `{object_id, body, provenance, cache_status, duration_ms}`. Plus `/healthz`, `/readyz`, `/docs/openapi.json` mirroring apps/api. 10 integration tests cover the contract, the two-kernel fleet-invalidation SLO (RFC §5.4 — both kernels' caches evict after a deploy event lands in the shared outbox), tenant isolation of invalidation, cursor monotonicity, and non-deploy events being ignored.
- **TASK-12 · Runtime API Auto-Derivation** — RFC §9.2 CRUD endpoints auto-derived from deployed Entity metadata. Migration 0004 adds the `ops` schema + `ops.entity_row(tenant_id, entity_id, row_id uuid, body jsonb, status, created_at, updated_at, deleted_at)` with strict-tenant RLS (no vendor escape hatch for ops data). `@erp/db` ships `EntityRowRepository extends TenantRepository` with `list/get/create/patch/softDelete` + a `createInTx` variant for callers that already hold a tenant-bound transaction (13 integration tests). `@erp/kernel-runtime` populates: `zodFromField(field)` translates every one of the 15 Field variants to a Zod validator; `materialize(resolved)` returns `{entity, fieldsByName, createValidator, patchValidator}` (strict keys, required-field enforcement on create, full-optional on patch); `MaterializedEntityCache` is a version-keyed LRU (24 unit tests). Apps/api adds: `PermissionGate` (RFC §13.1 level 1 — role-based entity-level grants, inheritance via `inherits_from`, deny-by-default), `RuntimeEntityService` (resolve → materialize → gate → validate → repository, returns `Result<T, RuntimeError>`), and `GET/POST/PATCH/DELETE /v1/:entity[/:id]` routes. Cache key hashes the full provenance stack so adding an L2 layer for a tenant rotates the key even when every layer's version is 1. Storage strategies `jsonb` and `hybrid` ship in Phase 1; `native` and `side_table` return 501 (CLAUDE.md §6 — deferred to when the Migration Primitives land). Permission Gate denies before leaking entity existence — an unknown entity to an unauthorized caller returns 403, not 404. 18 new integration tests cover CRUD contract, permission gate levels (deny-by-default, no-matching-role, action-not-granted), OpenAPI registration, and the CLAUDE.md §13 E2E: deploy ent.customer at L0 → deploy a Change Set adding loyalty_tier at L2 → POST a customer with the new field → GET it back — all without a process restart or redeploy. `apps/api/src/server.ts` wires the new service + gate + cache in the single buildServer() call. `MetadataObjectRepository.listActiveObjectIds(tenantId, type)` added for the gate to enumerate tenant-visible `prm.*` ids (vendor + tenant-scoped union).
- **TASK-13 · Reference Tenant Seed** — `pnpm db:seed` now populates `t_demo_retail` in three idempotent phases. Phase 1 (vendor L0): `ent.customer`, `ent.product`, `ent.invoice` Entities + `prm.admin` + `prm.viewer` Permissions, all with Arabic + English labels. Phase 2 (tenant L2 for `t_demo_retail`): Customer gets a `loyalty_tier` enum added at the tenant layer — demonstrates the five-layer customization model with real data. Phase 3 (rows): 50 Customers (deterministic mixed-locale names + phones + cities), 50 Products (localized_string names in Arabic and English across grocery/electronics/apparel/household), 50 Invoices linking back to the seeded Customers. Each phase is idempotent: `cs_seed_platform_v1` and `cs_seed_demo_retail_v1` marker Change Sets gate the metadata phases; row inserts use deterministic sha1-derived UUIDs + `ON CONFLICT (tenant_id, entity_id, row_id) DO NOTHING` for row-level idempotency. `scripts/seed.ts` uses `set_config('app.current_tenant', $1, true)` (the parameterizable form) rather than `SET LOCAL` so tenant ids flow through placeholders. 8 integration tests run the seeder against Testcontainers Postgres: first pass inserts exactly {5 vendor objects, 1 tenant object, 3 × 50 rows}; second pass is a strict no-op; partial-completion recovery (vendor phase succeeded but tenant phase didn't) fills only the missing phases; the Runtime API serves all 50 Customers with loyalty_tier populated, 50 Products with Arabic + English labels, 50 Invoices each referencing a seeded customer; `prm.viewer` can read but not create. Also added `packages/config/tsconfig.json` (shim extending the base) so vite/vitest's tsconfck resolver doesn't error when a test imports from `scripts/`.
- **Architectural decision records** — [ADR-0002](./docs/adr/0002-better-auth-zod-4-deferral.md) documents why Better Auth integration was deferred through TASK-10 and TASK-12 (peer-depends on Zod 4 while `@erp/core` is on Zod 3), and pins the Zod 3 → Zod 4 migration path for a follow-up TASK-10.1. Three alternatives (pin Better Auth at 1.0.x, write minimal custom auth, swap Zod for Valibot/ArkType) considered and rejected with reasoning recorded.
- **Audit log hash chain (RFC §13.2)** — `@erp/db` ships `AuditRepository` with `appendInTx(trx, input)` writing hash-chained rows inside caller-supplied transactions. Each row's `after_hash = sha256(before_hash || canonicalize(payload))`; per-tenant chains (null-tenant rows form their own vendor chain); a `pg_advisory_xact_lock(hash(tenant_id))` inside the append serializes concurrent writers so the chain never forks. `verifyChain(tenantId)` replays and returns `{audit_pk, reason}` on the first break — catches both tampered diffs (after_hash mismatch) and removed rows (before_hash mismatch). `canonicalize()` and `computeAfterHash()` are exported for offline tamper-detection sweeps. `EntityRowRepository` gains `patchInTx`/`softDeleteInTx` so the Runtime API service can run row mutations and audit appends in one transaction. `RuntimeEntityService.create/patch/delete` wrap the full flow in `withTenantContext()`; every successful `POST/PATCH/DELETE /v1/:entity[/:id]` now writes an audit row with `action = <entity_id>.<verb>`, `target_id = row_id`, diff (`{after}` / `{before,after,changed}` / `{before}`), and context (`{request_id, trace_id}`) — denied/failed requests write nothing. 14 new AuditRepository integration tests + 4 Runtime API audit assertions. Existing ChangeSetRepository audit writes (TASK-07) remain unchained for now; `readLastHashInTx` filters `WHERE after_hash IS NOT NULL` so they don't fork the new chain. Backfill tracked as a separate follow-up.
